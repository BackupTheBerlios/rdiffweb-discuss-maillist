On 11/21/06, Josh Nisly <rdiffweb at joshnisly.com> wrote:
> Essentially what both of you are wanting is a server-side backup where
> the server pulls from the client, instead of the client pushing to the
> server. Is that correct?

Yes

>
> I've long disliked this, because of the hassle of moving clients, dynmic
> IPs, etc. (BackupPC has this architecture, btw.) However, on an internal
> lan, where IPs are static, I could see some benefit. Another reason not
> to do this is that the core of rdiffWeb is platform independent, so
> (theoretically) you could run it on Windows. Put another way, most of
> rdiffWeb is Python, not shell scripting.

In the system I've written (which is all shell script, but I guess
could easily be ported to python), I used zeroconf for finding hosts.
i.e. hosta.local, hostb.local etc etc. Of course, this excludes
windows from the system, but as my system was designed for linux and
OS X, it works like a charm.

>
> One option might be to provide a "plugin" interface, where individual
> plugins could tie into to add pages (possibly linked to from the admin
> page.) This would allow us to create other modules of fringe benefit,
> such as tracking of total disk space used and billing, etc.
>
> Bottom line is that I'm busy enough that I probably won't be coding this
> anytime soon, but if someone else wants to code it in a clean, module
> form, I'll likely accept it.

Sounds good. Hopefully when I get some time, I'll get my hands dirty
and have a look at how rdiffWeb could do this. At the least, I'll code
up 2 pages, one to schedule backups via cron (mine also uses a client
"ping" program to force a backup now via "at"), and another page to
edit the list of backups.

I'll let you know how things go!
