Josh Nisly wrote:
> I think that there have been enough changes to rdiffWeb to call for 
> another release. As I'm wrapping up to release, I'm wondering if there 
> are any features that someone really wants to get into the next release. 
> If there are, I would consider trying to get them finished before 0.4

Thanks for your great application glad to hear it is still being
developed. A couple of simple ideas:

1. Any improvements you could make to the user management interface
would be great. Some of the buttons seem wrong ("edit" where it should
say "save" etc) and some icons would be nice ;-)  I quite like the
commandline "console" of applications like trac (which uses SQLite
instead of full on SQL server on the backend) since you can add users
easily.

2. One thing I notice when visiting http://rdiffwebhost/status/ is that
the backups listed under each date eg:

Successful Backups for 2007-09-24
* 2007-09-24 03:01:58 - 172.16.1.8/home - 1.37 GB

have no links to the repo. (e.g. the "172.16.1.8/home" could be a link
to: http://rdiffwebhost//browse/?repo=172.16.1.8%2Fhome&path=%2F in the
above example).

                           ++++++++++++++

Crazy deluxe ideas:

How about a little calendar widget in the UI to scroll back and forth
over time through the backups - not sure how that would work :-) perhaps
something like http://www.dynarch.com/demos/jscalendar/ ? mind you
there's some smarter UIs out there for dates using AJAX :)

What would be really cool is if rdiffweb were able to run a few simple
I/O and disk-status functions on the backup target to show "space
remaining" etc. I guess that may be quite a bit out of scope since
rdiffweb is more for browsing backup repositories and restoring files
that aren't "host dependent" than it is for cloning systems or managing
disks etc. It's just that with python you can do so many things :)
