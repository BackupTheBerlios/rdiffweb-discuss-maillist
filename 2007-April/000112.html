Thanks for your suggestions! A lot of these are things that I've wanted 
to do, but haven't had time. I've interspersed comments below.

Jesus Roncero wrote:
> Hi,
>
> As I said last week, here is a list of what I would like to do on rdiff-web, 
> or at least what we would like to use it. The idea we have in the office is 
> to have some kind of automated backup system for workstations where the admin 
> user creates an entry for each of the users and a nightly backup script will 
> retrieve the data from the workstations, possibly by having some remote 
> directories mounted or something. This doesn't necessarily need to be part of 
> rdiff-web but might be interesting to have it.
>
> So, here there is a list of things I've been thinking:
>
> - Add more logging, depending on whether you are running it in debug mode or 
> not. It is useful to have more information when you are debugging or you are 
> reading the source code to understand how it works.
>   
Not sure about logging at the source code level, but we need logging of 
page accesses, etc. CherryPy supports Apache-style logging, but we need 
a way to specify a log file and have accesses/errors go there. The most 
comment complaints are installation issues, and this would help with that.
> - The admin user manages the users. So, I'd like to extend the user admin 
> capabilities. For example, to add more parameters or others that might be 
> included in the future.
>   
Here is an architecture that I've been considering: all each user to be 
an "administrator", but only manage users within their root directory. 
That is, they could add/delete users, etc, but only within their directory.

We also need a way to generically store information about users, such as 
preferences. (Sort order, for example.) Is this what you are referring 
to here?
> - Separate users authentication from the "users" database, or have it as 
> something optional. So, we can have the users database where all the 
> information is stored, but use other backends for authentication, such as 
> other mysql dbs, or LDAP. This could help integration with existing 
> environments.
>   
I had initially wanted to support native linux user authentication, but 
couldn't find a clean, widely distributed python module that integrates 
with pam. The biggest goal for me with rdiff-web is keeping it easy to 
install and set up. If a user can't install and set up rdiff-web in 5-10 
minutes, then something is wrong, IMHO.

I'm all for separating the authentication and users list, though. I'll 
try to come up with a proposal for this sometime.
> - It would be possible that the user could choose their backup directories. 
> This would work with the backup script I just talked about, so they can 
> choose the directories (Need to think more about this)
>   
What I suggest is running "rdiff-web-config spiderrepos" via cron every 
15 minutes or so. (That's what I do.) That way all the user's repos show 
up. Is that what you're talking about?
> - Improvements to the browsing interface (nothing specific, just anything that 
> could be improved)
> - Improve the installation script.
> - Improve the status page (it fails when accessing it without credentials)
> - anything?
>   
I'd welcome suggestions on any of these.
> Well, those are the ideas floating in my mind. They don't necessarily need to 
> be included into trunk, but it would be nice to have them and, also, not have 
> something that is very different from trunk (a specific fork). So, that's 
> probably what I am going to start working on to have something similar to 
> what we want to have here :). Please, comments/objections/suggestions are 
> very welcome.
>
>   
I'd like to avoid a fork, if possible, as well. Right now, the time I 
have available to work on rdiff-web is pretty limited, but I'll see what 
I can do.

Thanks again,
JoshN
