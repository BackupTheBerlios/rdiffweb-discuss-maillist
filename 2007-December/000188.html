I'm working on making the setup process for rdiffWeb more streamlined. 
My current design is to have rdiffWeb display web-based setup after 
installation, negating the need for using the commandline. The plan is 
to require the root password to be entered to prevent unauthorized users 
from accessing setup.

Since the root password is going to be sent over the network, I don't 
want to run setup over the network unencrypted. I see two options: one, 
the setup screen could be bound to localhost, thus preventing data from 
traveling over the network. The other option is to generate SSL 
certificates on the fly, and run setup over HTTPS. (The linux router 
project smoothwall does something similar to this.) I'm planning to 
optionally generate SSL certs during the setup process anyway, but at 
install time, I don't know what hostname (i.e. CommonName) to use for 
the certificates. So the user would probably get two SSL certificate 
prompts when accessing setup, one because the cert is self-signed, and 
the other because the cert is for the wrong hostname.

Which would you as a user prefer? Would you prefer to only be able to 
set up the server from the same computer, or would you prefer to get 
warnings about the SSL certificates when you run setup?

Looking forward to your feedback,
JoshN
